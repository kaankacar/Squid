## 2024-05-20 - [Precompute transaction hash to avoid repeated XDR parsing]
**Learning:** Parsing Stellar XDR using `TransactionBuilder.fromXDR` is highly resource-intensive in Node.js. In `relayer/src/services/stellar.ts` and `defender.ts`, this was being done repeatedly inside a `for` loop in `getTransactionStatus` when searching the `pendingTransactions` queue.
**Action:** When a transaction is first added to the pending queue, compute and store its hash (`transaction.hash().toString('hex')`) in the queue object. Then, inside loops or lookup methods, just compare the precomputed hash instead of parsing the XDR again. This trades a tiny amount of memory for a significant CPU speedup during transaction polling.
## 2024-05-21 - [Parallelize Horizon API Calls to Hide XDR Parsing Latency]
**Learning:** `TransactionBuilder.fromXDR` is highly CPU-bound and can block the main thread for a non-trivial amount of time when estimating fees. When estimating fees, both `horizon.ledgers().call()` and `horizon.feeStats()` were being fetched sequentially, interspersed with the XDR parsing, accumulating overall wait time.
**Action:** Always dispatch independent network requests (like `ledgers` and `feeStats`) concurrently via `Promise.all`. Furthermore, trigger these asynchronous requests *before* executing synchronous, CPU-heavy operations (like XDR parsing). This effectively hides the CPU-bound operation's latency inside the network round-trip time, reducing total request time by removing the blocking nature of the previous sequential layout.

## 2024-05-22 - [Parallelize Multiple Smart Contract Queries]
**Learning:** In the agent's autonomous loop, operations like `scan()` often require multiple independent read-only calls to the smart contract (e.g., getting all agents, dead agents, and vulnerable agents). Executing these sequentially accumulates significant network latency, particularly with Stellar/Soroban RPC calls, causing slow and blocking ticks.
**Action:** Always identify independent network and smart contract queries within the same lifecycle method (such as `getAllAgents()`, `getDeadAgents()`, etc.) and dispatch them concurrently using `Promise.all()`. This hides the cumulative network latency of individual calls behind the longest single round-trip, noticeably speeding up the overall interval operation.
